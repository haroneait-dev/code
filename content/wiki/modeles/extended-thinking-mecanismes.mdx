Avec la **réflexion étendue**, le modèle raisonne en interne avant de produire sa réponse visible. Cela améliore la qualité sur les tâches complexes (debug, planification, math, code multi-fichiers), au prix de tokens supplémentaires. **Tous les modèles Claude actuels** la prennent en charge — mais **le mécanisme a changé en 2026**, et c'est le piège n°1 à connaître.

## ⚠️ Le mécanisme dépend du modèle

Historiquement, on activait la réflexion avec `thinking: {type: "enabled", budget_tokens: N}`. **Ce n'est plus le cas sur les modèles récents.**

| Modèle | Comment activer la réflexion |
|---|---|
| **Opus 4.8 / 4.7** | `thinking: {type: "adaptive"}` **uniquement**. `{type: "enabled", budget_tokens: N}` renvoie une **erreur 400** — `budget_tokens` est **supprimé** (tout comme `temperature`, `top_p`, `top_k`). |
| **Opus 4.6 / Sonnet 4.6** | `thinking: {type: "adaptive"}` (recommandé). `budget_tokens` y est **déprécié** mais encore fonctionnel (échappatoire transitoire). |
| **Anciens modèles** (Sonnet 4.5, etc.) | `thinking: {type: "enabled", budget_tokens: N}`, avec `budget_tokens` **strictement inférieur** à `max_tokens` (min 1024). |

En clair : avec **adaptive**, Claude décide lui-même **quand** et **combien** réfléchir — il n'y a plus de budget de tokens à régler à la main. Tu doses la profondeur avec le paramètre `effort` (ci-dessous).

```javascript
// Opus 4.8 — réflexion adaptative (la bonne façon en 2026)
{
  model: "claude-opus-4-8",
  max_tokens: 16000,
  thinking: { type: "adaptive" },
  output_config: { effort: "high" },   // low | medium | high | xhigh | max
  messages: [{ role: "user", content: "Démontre que…" }]
}
```

> Note : sur Opus 4.7/4.8, la réflexion adaptative est **désactivée par défaut** — une requête sans champ `thinking` tourne sans réflexion. Mets `thinking: {type: "adaptive"}` explicitement pour l'activer.

## Le paramètre `effort`

`effort` (GA, pas de header beta) contrôle la profondeur de réflexion **et** la dépense globale de tokens, via `output_config: {effort: …}` (à l'intérieur de `output_config`, pas au niveau racine).

| Niveau | Usage |
|---|---|
| `low` | Tâches courtes/cadrées, latence prioritaire |
| `medium` | Bon équilibre coût/qualité |
| `high` | Défaut (équivaut à l'omettre) ; minimum recommandé pour le travail exigeant |
| `xhigh` | **Meilleur réglage pour le code et l'agentique** (Opus 4.7/4.8) — entre `high` et `max` |
| `max` | Correction avant tout, coût secondaire — **Opus uniquement** (4.6+) |

`max` n'est pas disponible sur Sonnet ni Haiku ; `effort` lève une erreur sur Sonnet 4.5 / Haiku 4.5.

## Réflexion résumée (`thinking.display`)

Sur **Opus 4.7/4.8**, les blocs de réflexion sont renvoyés **vides par défaut** (`display: "omitted"`) — le nom du champ reste `block.thinking`, mais son texte est vide. Pour récupérer un **résumé** du raisonnement, opte explicitement :

```javascript
thinking: { type: "adaptive", display: "summarized" }
```

Si ton produit affiche le raisonnement à l'utilisateur, le défaut `"omitted"` ressemble à une longue pause avant la réponse : passe en `"summarized"` pour restaurer une progression visible. Ce résumé fournit l'essentiel du bénéfice intellectuel tout en limitant les usages abusifs.

## Coût : les tokens de thinking sont des output

Les tokens de réflexion sont **facturés comme des tokens de sortie**, au tarif output standard ([tarification](/wiki/tarification)). Une tâche qui génère 5 000 tokens de réflexion + 500 tokens de réponse coûte ~5 500 tokens de sortie. En production : **plafonne** via `max_tokens` (un défaut raisonnable : 3 à 5× la longueur de réponse visible attendue) et baisse l'`effort` d'un cran si la facture grimpe.

## Limites de sortie par modèle

| Modèle | Sortie max (standard) |
|---|---|
| Opus 4.8 / 4.7 / 4.6 | jusqu'à **128k** tokens |
| Sonnet 4.6 / Haiku 4.5 | jusqu'à **64k** tokens |

> Au-delà de ~16k tokens de sortie, **streame la réponse** (`.stream()` + `.get_final_message()` / `.finalMessage()`) pour éviter les timeouts HTTP du SDK. Sur l'API [Batch](/wiki/batch), un en-tête beta (`output-300k-…`) peut relever la limite à **300k** pour Opus 4.8/4.7/4.6 et Sonnet 4.6.

## Task Budgets (beta, Opus 4.7 / 4.8)

`output_config: {task_budget: {type: "tokens", total: N}}` indique au modèle combien de tokens il a pour **toute une boucle agentique** (réflexion + appels d'outils + réponse). Il voit un compte à rebours et s'auto-modère. C'est **distinct** de `max_tokens`, qui reste un plafond dur par réponse dont le modèle n'a pas conscience. Minimum `total` : **20 000 tokens** ; en-tête beta `task-budgets-2026-03-13`.

## Compatibilité

- **Streaming** : supporté — tu reçois des `thinking_delta` puis des `text_delta`.
- **Tool use** : supporté ; la réflexion adaptative active automatiquement la **réflexion entrelacée** (interleaved thinking) entre les appels d'outils, sans header beta.
- **Prompt caching** : les blocs de thinking ne portent pas de `cache_control` explicite mais sont mis en cache automatiquement ([prompt caching](/wiki/prompt-caching)).

## Erreurs fréquentes

- **Utiliser `budget_tokens` sur Opus 4.7/4.8** → 400. Passe à `thinking: {type: "adaptive"}` + `effort`.
- **Activer la réflexion partout** : explose la facture sans gain mesurable sur les tâches simples.
- **Renvoyer le `thinking` brut à l'utilisateur** : préfère `display: "summarized"` et ne logue le détail qu'en debug.
- **Oublier `max_tokens` adapté** à `xhigh`/`max` : prévois ≥ 64000 pour laisser de la place à la réflexion + l'action.
