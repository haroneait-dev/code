L'**extended thinking** (parfois appelé « thinking » ou « reasoning ») permet à Claude de produire une chaîne de pensée interne avant de générer sa réponse finale. Cette phase de réflexion est tokenisée, facturée comme des tokens de sortie, mais elle améliore notablement la qualité des réponses sur des tâches complexes — debugging, planification, math, code multi-fichier.

## Concept

Quand le thinking est activé, l'API renvoie deux types de blocs de contenu :

1. **`thinking`** : la réflexion interne du modèle, structurée en plusieurs étapes.
2. **`text`** : la réponse finale destinée à l'utilisateur.

Le modèle alloue dynamiquement des tokens dans une **enveloppe de réflexion** (le `budget_tokens`) avant de produire sa réponse.

## Activation côté SDK

```ts
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();

const res = await client.messages.create({
  model: "claude-sonnet-4-6",
  max_tokens: 16000,
  thinking: {
    type: "enabled",
    budget_tokens: 8000,
  },
  messages: [
    { role: "user", content: "Debug ce script Python qui freeze..." },
  ],
});

for (const block of res.content) {
  if (block.type === "thinking") {
    console.log("[REFLEXION]", block.thinking);
  } else if (block.type === "text") {
    console.log("[REPONSE]", block.text);
  }
}
```

En Python :

```python
res = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=16000,
    thinking={"type": "enabled", "budget_tokens": 10000},
    messages=[{"role": "user", "content": "..." }],
)
```

## Règles de budget

| Paramètre | Effet |
|---|---|
| `budget_tokens` | Plafond de tokens alloués à la réflexion |
| `max_tokens` | Total max output (thinking + réponse finale) |
| `budget_tokens` < `max_tokens` | Toujours, sinon erreur |

> Important : le `budget_tokens` est un **plafond souple**. Le modèle peut s'arrêter avant si la réflexion est suffisante. Mais il ne dépassera jamais ce budget.

## Facturation

Les tokens de thinking sont facturés au tarif **output tokens** du modèle utilisé. Concrètement :

| Modèle | Cas | Coût |
|---|---|---|
| Sonnet 4.6 | 5k tokens thinking + 1k réponse | 6k output tokens facturés |
| Opus 4.7 | 10k tokens thinking + 2k réponse | 12k output tokens facturés |

Un budget de 10 000 tokens sur Opus peut donc significativement gonfler la facture si tu l'actives sur toutes les requêtes.

## Quand activer le thinking

| Situation | Activer ? |
|---|---|
| Bug subtil multi-fichiers | Oui |
| Génération de plan d'action | Oui |
| Math, raisonnement formel | Oui |
| Q&A factuel court | Non |
| Reformulation, traduction | Non |
| Classification | Non |

## Choix du budget

| Tâche | Budget conseillé |
|---|---|
| Debug simple | 2 000 - 5 000 |
| Refactor moyen | 5 000 - 10 000 |
| Architecture, plan complexe | 10 000 - 25 000 |
| Recherche profonde | 25 000+ |

Commence bas, augmente seulement si la qualité ne suit pas.

## Compatibilité

- **Streaming** : supporté. Tu reçois des `thinking_delta` puis des `text_delta`.
- **Tool use** : supporté. Le modèle peut « penser » avant un appel d'outil puis après.
- **Prompt caching** : compatible. Les blocs `thinking` peuvent être inclus dans le cache.

```json
{
  "type": "message_delta",
  "delta": { "stop_reason": "end_turn" },
  "usage": {
    "input_tokens": 1200,
    "output_tokens": 850,
    "cache_read_input_tokens": 5000
  }
}
```

## Lecture du thinking en debug

Logger les blocs `thinking` aide énormément à comprendre **pourquoi** un agent a fait un choix donné. Ne les expose **jamais** à l'utilisateur final brut : la réflexion peut contenir des hypothèses intermédiaires erronées ou des informations sensibles.

```ts
if (block.type === "thinking") {
  logger.debug({ thinking: block.thinking });
}
```

## Erreurs fréquentes

- **Activer thinking partout** : explose la facture sans gain mesurable.
- **Budget trop bas** sur tâche complexe : le modèle tronque sa réflexion.
- **Renvoyer le `thinking` à l'utilisateur** : viole les UX guidelines + risque de fuites.
- **Oublier `max_tokens` ajusté** : tu reçois une réponse vide si `budget_tokens` consomme tout `max_tokens`.
