Les variables d'environnement offrent un contrôle programmatique de Claude Code, utile en CI, derrière un proxy d'entreprise, ou pour désactiver des comportements. Claude Code les lit **au démarrage** : un changement prend effet au prochain lancement de `claude`.

## D'où sont lues les variables

Plusieurs sources, par ordre de lecture :

1. L'environnement du **shell**
2. Un fichier `.env` à la racine du projet
3. `~/.claude/env`
4. Le champ [`env` de `settings.json`](/wiki/cli/settings-json) (et `claudeCode.environmentVariables` pour l'extension VS Code)

Le réglage `settings.json` s'applique **quand la variable n'est pas déjà définie** ailleurs. L'interaction exacte avec les flags varie : `--model` / [`/model`](/wiki/slash-commands) priment sur `ANTHROPIC_MODEL`.

## Choisir le modèle

```bash
export ANTHROPIC_MODEL=claude-sonnet-4-6
```

Override ponctuel par session : `claude --model …`. Voir [Modèles](/wiki/modeles).

## Authentification

| Variable | Rôle |
|---|---|
| `ANTHROPIC_API_KEY` | Clé d'API (compte Console) |
| `ANTHROPIC_AUTH_TOKEN` | Token OAuth (rempli après `claude login`) |
| `ANTHROPIC_BASE_URL` | URL d'API alternative (proxy, gateway interne) |

> ⚠️ `ANTHROPIC_API_KEY` **écrase** l'auth OAuth : si tu es connecté en Claude Max et qu'une vieille variable traîne, tu seras facturé sur la clé API. **Ne la commite jamais** dans un dépôt ni dans un `.vscode/settings.json` partagé — utilise les réglages utilisateur ou un secret CI.

## Désactiver des comportements

| Variable | Effet |
|---|---|
| `DISABLE_AUTOUPDATER=1` | Désactive la mise à jour automatique |
| `DISABLE_TELEMETRY=1` | Désactive la télémétrie |
| `DISABLE_AUTOCOMPACT=1` | Désactive la compaction automatique du contexte |
| `MAX_THINKING_TOKENS=0` | Coupe la réflexion étendue |

> La référence officielle liste l'ensemble des variables et leurs valeurs acceptées (plus de 200 selon les versions). Vérifie toujours la doc pour les valeurs exactes.

## Délais d'expiration & exécution

`API_TIMEOUT_MS` règle le délai des requêtes. Pour [Bash](/wiki/outils) : `BASH_DEFAULT_TIMEOUT_MS` et `BASH_MAX_TIMEOUT_MS`. Pour fixer une variable de façon durable sous Windows : `setx API_TIMEOUT_MS "1200000"` (puis nouveau terminal), ou sous PowerShell via `[Environment]::SetEnvironmentVariable(...)`.

## Proxy d'entreprise

En réseau corporate, configure avant de lancer Claude :

```bash
export HTTPS_PROXY=http://proxy.corp:8080
export NO_PROXY=api.anthropic.com,gateway.corp
```

Si ton organisation route le trafic Claude par une passerelle interne, pointe `ANTHROPIC_BASE_URL` vers cette passerelle et ajoute son hôte à `NO_PROXY` (pour éviter une double interception). Voir [Architecture & Enterprise](/wiki/enterprise).

## Observabilité (OpenTelemetry)

Claude Code supporte les variables OTel standard : `OTEL_METRICS_EXPORTER`, `OTEL_LOGS_EXPORTER`, `OTEL_EXPORTER_OTLP_ENDPOINT`, `OTEL_EXPORTER_OTLP_PROTOCOL`, `OTEL_EXPORTER_OTLP_HEADERS`, `OTEL_METRIC_EXPORT_INTERVAL`, `OTEL_RESOURCE_ATTRIBUTES`, etc. — pour envoyer métriques et logs vers ta stack d'observabilité.

## Variables injectées dans les sous-processus

Quand Claude exécute un outil `Bash`, un hook ou un MCP server, il **injecte** des variables décrivant l'état courant : `CLAUDE_PROJECT_DIR` (racine du projet), `CLAUDE_MODEL`, etc. Pratique pour qu'un hook écrive un log à la racine, peu importe d'où la commande est lancée :

```bash
#!/usr/bin/env bash
echo "$(date) — $1" >> "$CLAUDE_PROJECT_DIR/.claude/hooks.log"
```
