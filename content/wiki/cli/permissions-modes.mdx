Le mode de permission fixe le **comportement de base** de Claude Code face aux actions : qu'est-ce qui passe sans confirmation, qu'est-ce qui demande validation. On **superpose** ensuite des règles `allow`/`deny` ([settings.json](/wiki/cli/settings-json)) par-dessus ce baseline (sauf en `bypassPermissions`).

## Les 6 modes

| Mode | Comportement |
|---|---|
| **default** | Lit librement, mais demande confirmation pour les actions au-delà (écritures, commandes). |
| **acceptEdits** | Auto-approuve les **éditions de fichiers** pour la session ; tu ne valides que les commandes shell. Idéal pour itérer en relisant les diffs. |
| **plan** | Mode **recherche/planification** : Claude lit, analyse et propose un plan, **sans rien modifier**. |
| **auto** | Utilise un **classifieur de sécurité** : approuve les actions sûres, bloque les risquées. *Research preview.* |
| **dontAsk** | N'exécute **que** les outils pré-approuvés (allowlist) et refuse tout le reste. Bon pour l'automatisation « mains libres » plus sûre que bypass. |
| **bypassPermissions** | **Saute la couche de permission** (sauf écritures sur chemins protégés). À réserver aux environnements isolés. |

> Dans tous les modes **sauf** `bypassPermissions`, les écritures vers les **chemins protégés** (`.git`, `.claude`, fichiers rc du shell…) ne sont jamais auto-approuvées — garde-fou contre la corruption du dépôt et de la config de Claude.

## Changer de mode

- **En session** : `Shift+Tab` fait défiler `default → acceptEdits → plan` (le mode courant s'affiche dans la barre d'état). `auto` apparaît dans le cycle si ton compte y est éligible.
- **Au lancement** : `claude --permission-mode <mode>` (accepte `default`, `acceptEdits`, `plan`, `auto`, `dontAsk`, `bypassPermissions`) et l'emporte sur le `defaultMode` de tes [réglages](/wiki/cli/settings-json) pour ce run.
- **Persistant** : `"defaultMode": "acceptEdits"` dans `settings.json`.

`--dangerously-skip-permissions` est un raccourci pour `--permission-mode bypassPermissions` : le nom volontairement alarmant rend le compromis de sécurité explicite. À réserver aux pipelines CI/CD de confiance, en conteneur.

## Conditions du mode `auto`

`auto` n'est pas universel :

- **Plans** : Max, Team, Enterprise, API. **Pas** sur Claude Pro.
- **Modèle** : sur l'API Anthropic, Opus 4.6+ ou Sonnet 4.6 (pas Haiku ni claude-3).
- **Providers tiers** : sur [Bedrock, Vertex, Foundry](/wiki/enterprise/clouds), `auto` est désactivé tant que `CLAUDE_CODE_ENABLE_AUTO_MODE` n'est pas posé, et seul Opus 4.7/4.8 est pris en charge.
- **Team/Enterprise** : un admin doit l'activer.
- **En headless (`-p`)** : la session **avorte** après 3 blocages consécutifs ou 20 au total (pas d'humain pour valider) — `auto` est donc fragile face à du contenu adverse.

Les organisations peuvent ajuster ce qu'`auto` considère comme « infrastructure de confiance » via le bloc `autoMode` des réglages.

## Quel mode choisir ?

- **Pairing humain, session longue, relecture des diffs en fin** → `acceptEdits`.
- **Explorer/auditer sans rien casser** → `plan`.
- **Agent d'arrière-plan / CI fiable** → `dontAsk` + allowlist étroite (plus sûr qu'`auto` pour la prod).
- **Tâche jetable en conteneur isolé** → `bypassPermissions` (jamais sur ta machine principale).
- **Doute** → reste en `default`.

Complète toujours par le **sandbox** et des [hooks](/wiki/hooks) `PreToolUse` (qui s'exécutent **avant** la couche de permission et peuvent approuver/refuser/modifier un appel). Voir [Workflows & Sécurité](/wiki/workflows).
