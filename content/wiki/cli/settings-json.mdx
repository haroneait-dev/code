`settings.json` est le mécanisme officiel pour configurer le comportement de Claude Code : permissions, variables d'environnement, modèle, hooks, serveurs MCP. À distinguer du [`CLAUDE.md`](/wiki/demarrer/claude-md) qui, lui, contient des **instructions** (contexte), pas de la configuration.

## La hiérarchie (5 niveaux)

Claude Code lit plusieurs fichiers et les **fusionne**, du plus large au plus spécifique :

| Niveau | Emplacement | Usage |
|---|---|---|
| **Enterprise (managed)** | Politique gérée par l'organisation | Sécurité imposée — **priorité maximale** |
| **Utilisateur** | `~/.claude/settings.json` | Tes réglages, tous projets confondus |
| **Projet (partagé)** | `<projet>/.claude/settings.json` | Versionné, partagé avec l'équipe |
| **Projet (local)** | `<projet>/.claude/settings.local.json` | Tes overrides personnels (non versionné) |
| **Flags / env de session** | `--model`, variables d'env | Override ponctuel |

Claude Code ajoute automatiquement `.claude/settings.local.json` au `.gitignore`. **Règle clé** : les politiques Enterprise priment toujours ; pour le reste, le plus spécifique l'emporte. Une règle `deny` à un niveau ne peut pas être annulée par un `allow` à un niveau inférieur (ordre **deny > ask > allow**).

> Astuce : Claude Code crée des **sauvegardes horodatées** quand un fichier de config change (les 5 plus récentes par fichier). Un réglage cassé est récupérable sans archéologie Git.

## Autocomplétion avec `$schema`

Ajoute une ligne `$schema` pour obtenir l'autocomplétion de toutes les options dans ton éditeur :

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": ["Bash(npm run lint)", "Bash(npm run test:*)"],
    "deny": ["Read(./.env)", "Read(./secrets/**)"]
  }
}
```

## Le bloc `permissions`

C'est le réglage le plus important. Il définit les [outils](/wiki/outils) autorisés/refusés sans confirmation. Format : `Tool` ou `Tool(spécifieur)`.

```json
{
  "permissions": {
    "allow": ["Read", "Glob", "Grep", "Bash(git:*)", "Bash(npm:*)"],
    "deny": ["Bash(rm:*)", "Read(./.env)", "Read(**/*.pem)"]
  }
}
```

Les patterns suivent une syntaxe **glob** pour les chemins et un format `Tool(argument)` pour les commandes ; le suffixe `:*` autorise toutes les sous-commandes. À gérer aussi en session avec [`/permissions`](/wiki/slash-commands). Voir [modes de permission](/wiki/cli/permissions-modes) pour le baseline appliqué par mode.

## Le champ `env`

`settings.json` peut injecter des variables d'environnement dans **toutes** les commandes [Bash](/wiki/outils) et les scripts de [hooks](/wiki/hooks), au démarrage de la session :

```json
{
  "env": {
    "NODE_ENV": "development",
    "DATABASE_URL": "postgresql://localhost:5432/myapp"
  }
}
```

`env` est un `Record<string, string>` — les valeurs doivent être des chaînes (pas d'évaluation dynamique). Elles fusionnent avec ton shell, `settings.json` ayant priorité. Pour la liste complète des variables reconnues, voir [variables d'environnement](/wiki/cli/variables-env).

## Sélection de modèle

Tu peux fixer le [modèle](/wiki/modeles) par défaut dans `settings.json`. Les variables `ANTHROPIC_MODEL` / `CLAUDE_MODEL` et le flag `--model` (ou [`/model`](/wiki/slash-commands)) l'emportent sur ce réglage — pratique pour utiliser Sonnet en CI (coût) et Opus en local (qualité).

## Hooks et MCP

`settings.json` déclare aussi les [hooks](/wiki/hooks) (`PreToolUse`, `PostToolUse`, `UserPromptSubmit`, `Stop`, `SessionStart`…) et les serveurs [MCP](/wiki/mcp). Les outils MCP exposés sont préfixés `mcp__<serveur>__<outil>`.

## Autres réglages utiles

Selon les versions, `settings.json` expose des dizaines d'options : `defaultMode` (mode de permission par défaut), signature des commits Git, profondeur de réflexion, préférences de langue, et — côté Enterprise — `disableBypassPermissionsMode` pour bloquer entièrement le mode `bypassPermissions`.

## Bonnes pratiques

- **Versionne `.claude/settings.json`** pour partager la config d'équipe.
- **Ne commite jamais `settings.local.json`** : approbations perso et éventuels secrets dans `env`.
- Préfère des patterns **granulaires** (`Bash(git status)`) à un `Bash(*)` global.
- Le skill `fewer-permission-prompts` génère une allowlist pertinente à partir de tes transcripts.
