Une slash command custom, c'est juste un fichier Markdown. Pas de SDK, pas de plugin, pas de build : un `.md` dans le bon dossier, et la commande apparaît automatiquement dans `/help`. C'est probablement la fonctionnalité la plus rentable de Claude Code : chaque procédure que vous répétez peut devenir une commande, partagée avec votre équipe via git.

## Emplacement : projet ou utilisateur

Deux dossiers possibles, deux portées :

- `.claude/commands/` à la racine du projet : commande **partagée** avec l'équipe via git
- `~/.claude/commands/` dans votre home : commande **personnelle**, disponible dans tous vos projets

Si une commande existe aux deux endroits avec le même nom, la version projet l'emporte. C'est ce qui permet à une équipe de surcharger une commande globale par une version spécifique au repo.

## Structure minimale

Le strict minimum, c'est un fichier `.md` avec un titre et des instructions en français (ou la langue de votre choix) :

```md
---
description: "Lance les tests unitaires et affiche un résumé"
---

Exécute `npm test` et résume les résultats :
- nombre de tests passants / échouants
- temps total
- liste des fichiers en échec si applicable
```

Sauvegardé sous `.claude/commands/test.md`, ce fichier ajoute la commande `/test` à votre session. Le contenu du Markdown devient le prompt envoyé à Claude quand vous tapez `/test`.

## Frontmatter complet

Le frontmatter YAML accepte plusieurs champs qui modifient le comportement de la commande :

```md
---
description: "Audit sécurité de la PR courante"
argument-hint: "[numéro de PR optionnel]"
allowed-tools: [Read, Grep, Bash(gh pr view:*)]
model: claude-opus-4-7
---

Tu es un auditeur sécurité.

PR à auditer : $ARGUMENTS

1. Récupère le diff avec `gh pr diff`
2. Liste les vulnérabilités par sévérité
3. Propose un patch pour chaque problème HIGH
```

- `description` : affichée dans `/help`
- `argument-hint` : hint qui apparaît dans l'auto-complétion
- `allowed-tools` : restreint les tools que Claude peut utiliser pour cette commande
- `model` : force un modèle spécifique (utile pour les audits coûteux où vous voulez Opus)

## Exemple complet : commande de release

Voici une commande utile dans la vraie vie, `.claude/commands/release.md` :

```md
---
description: "Prépare une release : changelog, version, tag"
argument-hint: "<patch|minor|major>"
allowed-tools: [Read, Edit, Bash(npm version:*), Bash(git log:*), Bash(git tag:*)]
---

Prépare une release `$ARGUMENTS` :

1. Lis le dernier tag avec `git describe --tags --abbrev=0`
2. Liste les commits depuis ce tag avec `git log <tag>..HEAD --oneline`
3. Génère un CHANGELOG.md formaté en sections (feat, fix, chore)
4. Bumpe la version avec `npm version $ARGUMENTS --no-git-tag-version`
5. Affiche un résumé et demande confirmation avant de tagger
```

Usage :

```bash
> /release minor
Reading last tag: v1.2.3
Found 14 commits since v1.2.3:
  - feat: add wiki vote endpoint
  - fix: RLS policy on tips
  - chore: bump deps
Generated CHANGELOG.md
Version bumped to 1.3.0
Ready to tag? (y/n)
```

## Tester votre commande

Une fois le fichier créé, lancez :

```bash
> /help
... liste ...
/release   Prépare une release : changelog, version, tag

> /release patch
```

Si la commande n'apparaît pas, vérifiez :

1. Le fichier est bien dans `.claude/commands/` (et pas dans un sous-dossier sans intention de namespacing)
2. L'extension est bien `.md`
3. Le frontmatter YAML est valide (pas de tab, indentation correcte)
4. Vous avez bien rechargé la session (`/clear` ou redémarrage)

## Déploiement en équipe

Pour partager une commande, c'est `git add .claude/commands/release.md` et vous poussez. Les collègues qui pullent récupèrent la commande automatiquement, sans installation. Pas de plugin, pas de bundle, pas de versioning séparé : c'est ça la force du système.

Pensez à ajouter une ligne à votre `CLAUDE.md` pour documenter les commandes maison :

```md
## Slash commands custom
- /release <patch|minor|major>  — release helper
- /audit-sec                    — audit sécurité maison
- /db-migrate                   — génère et applique une migration Supabase
```
