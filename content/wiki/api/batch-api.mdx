La **Message Batches API** d'Anthropic permet d'envoyer jusqu'à **100 000 requêtes** dans un seul batch, traitées de manière asynchrone en moins de 24 heures (souvent quelques minutes). Le tarif est **réduit de 50%** par rapport aux appels synchrones, ce qui en fait l'outil parfait pour les workloads non temps-réel : génération de contenu en masse, annotation de datasets, évaluations, classification d'archives, enrichissement de bases de données.

## Quand utiliser le Batch ?

Posez-vous une seule question : **avez-vous besoin de la réponse dans la seconde ?** Si non, le batch est presque toujours le bon choix.

- Génération nocturne de résumés (newsletters, rapports).
- Évaluation d'un nouveau prompt sur 10 000 exemples.
- Migration : reclassifier toute une base existante.
- Enrichissement SEO : reformuler des milliers de fiches produit.

## Créer un batch

Chaque requête du batch est une `MessageCreateParams` standard, encapsulée dans un objet avec un `custom_id` unique pour retrouver le résultat plus tard.

```python
import anthropic
from anthropic.types.messages.batch_create_params import Request

client = anthropic.Anthropic()

batch = client.messages.batches.create(
    requests=[
        Request(
            custom_id=f"item-{i}",
            params={
                "model": "claude-sonnet-4-6",
                "max_tokens": 512,
                "messages": [{"role": "user", "content": f"Résume : {text}"}],
            },
        )
        for i, text in enumerate(my_documents)
    ]
)
print(batch.id, batch.processing_status)
```

## Suivre le statut

Le batch passe par `in_progress` puis `ended`. Le champ `request_counts` détaille `processing`, `succeeded`, `errored`, `canceled`, `expired`.

```python
batch = client.messages.batches.retrieve(batch.id)
print(batch.processing_status)
print(batch.request_counts)
# RequestCounts(processing=0, succeeded=9987, errored=13, canceled=0, expired=0)
```

Pour les pipelines, polling toutes les 30 s à 2 min suffit largement.

```ts
while (true) {
  const b = await client.messages.batches.retrieve(batchId);
  if (b.processing_status === "ended") break;
  await new Promise(r => setTimeout(r, 60_000));
}
```

## Récupérer les résultats

Une fois `ended`, les résultats se streament en JSONL via `.results()`. Chaque ligne contient le `custom_id` et le résultat (success ou error).

```python
for entry in client.messages.batches.results(batch.id):
    if entry.result.type == "succeeded":
        msg = entry.result.message
        save(entry.custom_id, msg.content[0].text)
    else:
        log_error(entry.custom_id, entry.result)
```

Format JSONL d'une ligne :

```json
{
  "custom_id": "item-42",
  "result": {
    "type": "succeeded",
    "message": { "id": "msg_...", "content": [{"type":"text","text":"..."}], "usage": {...} }
  }
}
```

## Limites et bonnes pratiques

- **24 h max** d'exécution : au-delà, les requêtes restantes passent en `expired`.
- **100 000 requêtes** ou **256 Mo** par batch — divisez si nécessaire.
- Les résultats restent disponibles **29 jours**.
- **Compatible** avec le tool use, la vision, le prompt caching (économies cumulables : -50% batch × -90% cache).
- Pour annuler : `client.messages.batches.cancel(batch_id)` — les requêtes en cours terminent, les autres sont annulées.
