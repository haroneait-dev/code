L'API **Message Batches** traite de nombreuses requêtes [Messages](/wiki/api) de façon **asynchrone et concurrente**, à **50 % du prix standard**. C'est l'outil idéal pour les charges non urgentes : classification, génération en masse, enrichissement de données, évaluations.

## La remise de 50 %

Toute l'utilisation d'un batch est facturée à **la moitié** des prix standard de l'API, sur l'input **et** l'output. La remise est **inconditionnelle** : pas de seuil de volume à atteindre. Elle se cumule avec le [prompt caching](/wiki/prompt-caching).

## Délai et limites

- Les batchs sont traités **sous 24 heures** (souvent quelques minutes à 1–2 h en pratique). Au-delà de 24 h, les requêtes restantes passent en `expired`.
- **100 000 requêtes** ou **256 Mo** par batch — divise si nécessaire.
- Les résultats restent disponibles **29 jours** après création.
- Comme un batch peut dépasser 5 minutes, privilégie la durée de cache **1 heure** si tu utilises le caching, pour de meilleurs taux de hit.

## Structure d'une requête

Un Message Batch est une **liste de requêtes**. Chaque requête comprend :

- un **`custom_id`** unique pour identifier la requête au retour (1 à 64 caractères, motif `^[a-zA-Z0-9_-]{1,64}$`) ;
- un objet **`params`** avec les paramètres standard de l'API Messages.

```javascript
const batch = await client.messages.batches.create({
  requests: [
    { custom_id: "ticket-001",
      params: { model: "claude-haiku-4-5", max_tokens: 256,
                messages: [{ role: "user", content: "Classe ce ticket : …" }] } },
    { custom_id: "ticket-002",
      params: { model: "claude-haiku-4-5", max_tokens: 256,
                messages: [{ role: "user", content: "Classe ce ticket : …" }] } }
  ]
});
console.log(batch.id, batch.processing_status);
```

## Suivre le statut, récupérer les résultats

Le batch passe par `in_progress` puis `ended`. Le champ `request_counts` détaille `processing`, `succeeded`, `errored`, `canceled`, `expired`. On **interroge** (poll) jusqu'à complétion — un intervalle de 30 s à 2 min suffit.

```python
while True:
    batch = client.messages.batches.retrieve(batch.id)
    if batch.processing_status == "ended":
        break
    time.sleep(60)

for entry in client.messages.batches.results(batch.id):
    if entry.result.type == "succeeded":
        save(entry.custom_id, entry.result.message.content[0].text)
    else:
        log_error(entry.custom_id, entry.result)   # errored | expired | canceled
```

Chaque résultat est rattaché à son `custom_id` ; loguer systématiquement `custom_id` ↔ `message.id` simplifie le rapprochement comptable et le debug.

## Paramètres non supportés

Un petit nombre de paramètres de l'API Messages ne sont **pas** acceptés en batch ; les inclure renvoie une erreur de validation. Vérifie la doc officielle pour la liste à jour avant de bâtir ton pipeline.

## Combiner avec le prompt caching

Les remises [caching](/wiki/prompt-caching) et batch **se cumulent** : l'effet combiné peut faire tomber le coût effectif autour d'un dixième du tarif non-caché. Comme le traitement est asynchrone, les hits de cache sont best-effort. Pour les maximiser : blocs `cache_control` **identiques** dans chaque requête, contexte partagé maximal, durée de cache **1 heure**, et flux de requêtes régulier.

## Quand utiliser le Batch ?

- ✅ Traitements **hors ligne** et **non urgents** (rapports nocturnes, ré-étiquetage massif, génération de descriptions, évaluations sur 10 000 exemples).
- ✅ Gros volumes où le coût prime sur la latence.
- ❌ Réponses **temps réel** (chatbot interactif) → reste sur l'API Messages standard avec le streaming.
