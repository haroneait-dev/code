Le **prompt caching** met en cache des portions de prompt réutilisées (gros contexte système, documents, exemples) pour que les requêtes suivantes qui partagent ce préfixe soient **plus rapides et moins chères**. C'est l'une des optimisations les plus efficaces de l'[API](/wiki/api).

## L'invariant à retenir : c'est un préfixe

Le cache est un **match de préfixe**. La clé est dérivée des **octets exacts** du prompt rendu jusqu'au bloc marqué `cache_control` (le « breakpoint ») inclus. **Un seul octet qui change quelque part dans le préfixe invalide tout ce qui suit.**

L'ordre de rendu est **`tools` → `system` → `messages`**. Un breakpoint sur le dernier bloc `system` met donc en cache `tools` + `system` ensemble. Conséquence pratique : place le **contenu stable d'abord** (prompt système figé, liste d'outils déterministe) et le **contenu volatil après** le dernier breakpoint (horodatages, identifiants par requête, la question qui varie).

## Automatique vs explicite

- **Automatique** : passe `cache_control: {type: "ephemeral"}` au niveau de la requête — le cache se pose sur le dernier bloc cacheable. La façon la plus simple, sans placer manuellement de breakpoints.
- **Explicite** : tu marques un bloc précis avec `cache_control` pour contrôler le point de césure (utile pour un préfixe partagé suivi d'une question variable).

Pris en charge sur **tous les modèles Claude actifs**. **Maximum 4 breakpoints** `cache_control` par requête.

```python
response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": LONG_CONTEXTE_PARTAGE,        # gros, stable
            "cache_control": {"type": "ephemeral"}
        }
    ],
    messages=[{"role": "user", "content": "La question qui varie..."}],
)
```

## Les multiplicateurs de prix

Le cache introduit une structure de prix par rapport au prix d'entrée de base :

| Opération | Multiplicateur |
|---|---|
| Écriture cache **5 minutes** (TTL par défaut) | **1,25×** le prix d'entrée de base |
| Écriture cache **1 heure** (`ttl: "1h"`) | **2×** |
| **Lecture** depuis le cache | **0,1×** (≈ 90 % d'économie) |

Ces multiplicateurs **se cumulent** avec d'autres modificateurs (remise [Batch](/wiki/batch), résidence des données).

**Seuil de rentabilité.** Avec le TTL 5 min, deux requêtes suffisent à rentabiliser (1,25× + 0,1× = 1,35× contre 2× sans cache). Avec le TTL 1 h, il faut au moins trois requêtes (2× + 0,2× = 2,2× contre 3×). Le TTL 1 h garde l'entrée vivante malgré des trous dans un trafic en rafales, mais l'écriture doublée demande plus de lectures pour être amortie.

## Le préfixe minimal cacheable (gotcha 2026)

En dessous d'une certaine taille, **le préfixe ne se met pas en cache — sans erreur**, juste `cache_creation_input_tokens: 0`. Et ce seuil **dépend du modèle** :

| Modèle | Préfixe minimal |
|---|---:|
| Opus 4.8 / 4.7 / 4.6 / 4.5, Haiku 4.5 | **4096 tokens** |
| Sonnet 4.6, Haiku 3.5 | **2048 tokens** |
| Sonnet 4.5 / 4 / 3.7 | **1024 tokens** |

Piège concret : un prompt de 3000 tokens **se cache sur Sonnet 4.5 mais pas sur Opus 4.8**. Si tu changes de modèle sans revoir la taille du préfixe, ton cache peut silencieusement cesser de fonctionner.

## Vérifier que ça marche

L'objet `usage` de la réponse rapporte l'activité de cache :

| Champ | Sens |
|---|---|
| `cache_creation_input_tokens` | Tokens écrits dans le cache (premium d'écriture ~1,25×) |
| `cache_read_input_tokens` | Tokens servis depuis le cache (payés ~0,1×) |
| `input_tokens` | Tokens traités plein tarif (le **reste non caché** uniquement) |

> ⚠️ `input_tokens` n'est **que** le reste non caché. La taille totale du prompt = `input_tokens + cache_creation_input_tokens + cache_read_input_tokens`. Si ton agent a tourné des heures mais que `input_tokens` affiche 4 K, le reste venait du cache : regarde la somme, pas le champ seul.

Si `cache_read_input_tokens` reste à **zéro** sur des requêtes répétées au préfixe identique, un **invalidateur silencieux** est à l'œuvre.

## Les invalidateurs silencieux

À traquer dans tout ce qui alimente le préfixe :

- `datetime.now()` / `Date.now()` / `time.time()` interpolé dans le prompt système → le préfixe change à chaque requête.
- `uuid4()` / identifiants de requête placés tôt dans le contenu → idem.
- `json.dumps(d)` **sans** `sort_keys=True`, ou itération d'un `set` → sérialisation non déterministe, les octets diffèrent.
- ID de session/utilisateur interpolé dans le prompt système → préfixe par utilisateur, aucun partage possible entre utilisateurs.
- **Changer les `tools` ou le `model` en cours de conversation** → les outils sont rendus en position 0, et les caches sont indexés par modèle : tout est invalidé.

> Le cache est scopé par **organisation + clé d'API + modèle** : pas de partage entre clés.

## Interaction avec l'extended thinking

Les blocs de réflexion ([extended thinking](/wiki/extended-thinking)) ne peuvent **pas** porter de `cache_control` explicite, mais ils sont **mis en cache automatiquement** comme partie du contenu lors des appels suivants (typiquement quand tu renvoies les blocs de thinking avec des résultats d'outils pour continuer). Lus depuis le cache, ils comptent comme tokens d'entrée.

## Interaction avec le Batch

Les remises [Batch](/wiki/batch) et caching **se cumulent**. Comme les requêtes batch sont asynchrones, les hits de cache sont « best-effort ». Pour les maximiser : utilise la durée **1 heure**, garde des blocs `cache_control` **identiques** dans chaque requête, et maintiens un flux régulier de requêtes.

## Bonnes pratiques

- Mets en cache ce qui est **stable et volumineux** (prompt système long, documents de référence) et place le breakpoint **après**.
- Garde le prompt système **figé** : injecte le contexte dynamique plus loin dans `messages`, pas dans le `system`.
- Sérialise les outils de façon **déterministe** (tri par nom) et ne les change pas en cours de route.
- Choisis la durée selon la fréquence de réutilisation (5 min pour des rafales rapprochées, 1 h pour des traitements espacés).
- Surveille `cache_read_input_tokens` pour vérifier l'efficacité réelle.
