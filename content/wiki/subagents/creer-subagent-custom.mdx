Un subagent, c'est **du Markdown** : le frontmatter YAML est la configuration, le corps est le prompt système. Pas de build, pas de SDK, pas de registre. Pour les fondamentaux (rôle de l'outil `Task`, parallélisme), voir l'article [Subagents](/wiki/subagents) ; ici on entre dans la définition concrète.

## Les agents intégrés

Claude Code embarque des subagents invoqués automatiquement :

- **Explore** — lecture seule, tourne par défaut sur **Haiku** (rapide et économique) : cherche et comprend une base de code sans la modifier.
- **Plan** — rassemble le contexte avant que Claude propose une stratégie en [mode plan](/wiki/cli/permissions-modes).
- **General-purpose** — gère les tâches mêlant exploration et modification.

Tu n'as pas à les appeler : Claude route vers eux selon ta demande.

## Structure d'un agent personnalisé

Place un fichier `.md` dans :

- `.claude/agents/` (**projet** — priorité la plus haute, versionnable)
- `~/.claude/agents/` (**utilisateur**)

En cas de même nom, le **projet** gagne. Si tu édites un fichier directement sur disque, **redémarre la session** ; les agents créés via l'UI [`/agents`](/wiki/slash-commands) prennent effet immédiatement.

```markdown
---
name: code-reviewer
description: "À utiliser pour relire les changements récents : sécurité, style, bonnes pratiques. MUST BE USED après un git diff important."
tools: Read, Grep, Glob          # optionnel ; omis → hérite des outils du thread
model: sonnet                    # optionnel ; ou 'inherit'
permissionMode: default
---

Tu es un ingénieur sécurité applicative senior. Quand on t'invoque :
1. Lance `git diff HEAD` pour voir les changements.
2. Identifie les zones à risque (auth, entrées, exposition de données).
3. Cherche : injection SQL, XSS, IDOR, contrôles d'accès manquants, secrets en clair.
4. Rends les conclusions en CRITICAL / HIGH / MEDIUM / LOW avec numéros de ligne.
```

Le subagent ne reçoit **que** ce prompt système (plus quelques détails d'environnement comme le dossier courant), pas tout le prompt système de Claude Code, ni le `CLAUDE.md` du projet de façon implicite.

## Le champ `description` fait tout

C'est lui qui décide de l'**auto-délégation**. Sois spécifique et riche en mots-clés ; les formules type « MUST BE USED for… » remontent la priorité. Une description vague est ignorée : Claude prend le premier agent dont la description matche l'intention. Un bon `description` contient un **déclencheur** (« après un commit », « quand le build casse ») et un **livrable** (« retourne un rapport », « propose un patch »).

## `tools` : moindre privilège

Si tu omets `tools`, l'agent **hérite** des [outils](/wiki/outils) du thread (y compris MCP). Restreins quand tu veux un contrôle serré :

- un reviewer obtient `Read + Grep` (pas `Bash` ni `Edit`) ;
- un rédacteur/migrateur `Read + Write + Edit` (pas `WebFetch`) ;
- un agent de recherche externe `WebSearch + WebFetch` (surtout pas `Edit`).

Cette allowlist sert deux objectifs : **sécurité** (limiter les dégâts) et **focus** (l'agent ne se disperse pas).

## Résolution du modèle

Claude Code résout le [modèle](/wiki/modeles) de l'agent dans cet ordre : la variable `CLAUDE_CODE_SUBAGENT_MODEL` si définie, puis le champ `model` du frontmatter, etc. La valeur `inherit` reprend le modèle de la session.

## Coût : attention aux tokens

Les subagents utilisent le **même modèle et les mêmes tokens** qu'une session normale, simplement dans leur **propre fenêtre de contexte** (pas de facturation séparée). L'intérêt : garder le thread principal propre — au lieu de lire 30 fichiers dans la conversation principale, on délègue (« trouve tous les appels à `loadUser` et résume »). Le coût : un workflow très orienté subagents peut consommer **~7× plus de tokens** qu'un thread unique, car chaque agent maintient son contexte. La compaction automatique du thread principal se déclenche d'ailleurs autour de 95 % de capacité.

## Patterns d'équipe

Chaîne des agents spécialisés via les [hooks](/wiki/hooks) `SubagentStop`/`Stop` : PM (spec) → Architecte (validation) → Implémenteur/Testeur (build & tests) → QA. Sérialise les étapes à risque, parallélise les sûres, garde un humain dans la boucle pour approuver les transitions sensibles.

## Trois façons de créer un subagent

1. **UI interactive** [`/agents`](/wiki/slash-commands) (la plus rapide ; Anthropic la recommande, et elle génère un premier squelette à partir d'une intention en langage naturel).
2. **Fichiers Markdown** dans `.claude/agents/` (versionnés dans Git → `git pull` partage l'agent avec toute l'équipe).
3. **Programmatique** via l'**Agent SDK** (paramètre `agents` de `query()`).
