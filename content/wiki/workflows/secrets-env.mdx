Un assistant IA qui lit ton repo lit aussi tes secrets. Si tu ne mets pas de garde-fous, ils finissent dans des prompts envoyés à l'API, dans des commits, ou pire, dans des screenshots partagés sur Slack. La défense est multi-couches : on combine `.gitignore`, deny patterns Claude, hooks `PreToolUse` et rotation régulière.

## Cloisonner les secrets dès la racine

Première règle : aucun secret dans le code, jamais. Tous tes secrets vivent dans `.env`, `.env.local`, ou un coffre (1Password, Doppler, Vercel env, AWS Secrets Manager). Ton `.gitignore` doit contenir :

```
.env
.env.local
.env.*.local
*.pem
*.key
credentials.json
secrets/
```

Vérifie régulièrement avec `git check-ignore .env` et fais un audit avec `git log -p | grep -i "api_key\|secret\|password"`. Un secret commité une fois reste dans l'historique : il faut le révoquer immédiatement, pas juste le supprimer du dernier commit.

## Configurer les deny patterns Claude

Dans `.claude/settings.json`, déclare explicitement les chemins interdits à la lecture :

```json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./**/credentials.json)",
      "Read(./**/*.pem)",
      "Read(./**/*.key)",
      "Bash(cat .env*)",
      "Bash(printenv*)"
    ]
  }
}
```

Claude refusera de lire ces fichiers même si tu le lui demandes par accident. C'est ta première ligne de défense contre le « tiens, montre-moi ce qu'il y a dans .env pour debug ».

## Mettre en place un hook PreToolUse

Les deny patterns couvrent les chemins connus. Pour les contenus dynamiques (un dump SQL avec des hash bcrypt, un log avec un token JWT), il faut intercepter à l'exécution. Ajoute un hook dans `settings.json` :

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash|Write|Edit",
        "command": "./scripts/scan-secrets.sh"
      }
    ]
  }
}
```

Le script scanne l'input du tool pour des patterns de secrets (regex sur `sk-`, `ghp_`, `xoxb-`, format JWT, etc.) et renvoie un exit code non-zéro pour bloquer l'opération. Tu peux t'appuyer sur `gitleaks` ou `trufflehog` en mode CLI.

## Éviter les fuites via la stack

Les secrets fuient souvent ailleurs que dans le code :

- **Logs de dev** : ne logue jamais un objet `process.env` entier
- **Erreurs serveur** : masque les stack traces qui contiennent des connection strings
- **Tests** : utilise des fixtures avec des fausses valeurs, jamais des copies de prod
- **Captures d'écran** : floute systématiquement avant de partager

Côté Claude, refuse de coller des extraits de prod dans le chat. Demande plutôt « voici la structure du payload » et reconstruit un exemple anonymisé.

## Rotation et détection a posteriori

Considère que tout secret a une espérance de vie limitée. Mets en place :

- Rotation automatique mensuelle pour les clés API critiques
- Scan continu du repo via GitHub Secret Scanning ou Gitleaks en CI
- Alerte Slack en cas de match positif
- Procédure documentée de révocation (qui prévenir, quels services redéployer)

Si Claude a accédé à un secret par erreur, considère-le compromis : révoque, fais tourner, audite les logs API. Ne pars jamais du principe que « c'était juste pour debug ».

## Pour aller plus loin

- Active la signature commit (`git config commit.gpgsign true`) pour tracer qui a poussé quoi
- Utilise un gestionnaire comme `direnv` pour charger les `.env` sans les exposer au shell parent
- Mets en place un pre-commit `gitleaks protect --staged` pour bloquer en amont
- Forme l'équipe : un secret leaké est rarement un bug technique, c'est un raté de process
