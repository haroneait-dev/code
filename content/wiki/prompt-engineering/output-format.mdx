Quand Claude alimente une chaîne de traitement automatisée, la sortie doit être **parsable à 100 %**. Une virgule en trop, un préambule *"Bien sûr, voici…"*, un bloc Markdown autour du JSON, et tout casse. Cet article couvre les patterns qui rendent les sorties Claude fiables en production.

## Le problème : Claude est conversationnel par défaut

Sans instruction explicite, Claude a tendance à :

- Ajouter un préambule poli (*"Voici le JSON demandé :"*)
- Envelopper le JSON dans ```` ```json ````
- Ajouter un commentaire de fin (*"N'hésitez pas si…"*)
- Inventer des champs non demandés

Ces comportements sont incompatibles avec un parsing strict. Trois familles de solutions : **instructions explicites**, **prefilling**, et **tool use**.

## Solution 1 : instructions explicites + schéma

Décrivez précisément le schéma attendu, avec un exemple.

```text
Tu réponds UNIQUEMENT avec un objet JSON valide, sans Markdown,
sans préambule, sans commentaire. Schéma attendu :

{
  "intent": "string (l'un de : 'question', 'commande', 'plainte')",
  "confidence": "number entre 0 et 1",
  "entities": ["string"]
}

Si tu ne peux pas répondre, renvoie : {"error": "raison courte"}
```

Cette approche fonctionne dans 90 % des cas avec Sonnet 4. Pour atteindre 99 %, passez au prefilling.

## Solution 2 : prefilling (la technique la plus puissante)

Le prefilling consiste à **commencer la réponse de Claude vous-même**. Le modèle continue à partir de votre amorce. C'est mécaniquement impossible pour lui d'ajouter un préambule.

```json
{
  "model": "claude-sonnet-4-6",
  "messages": [
    { "role": "user", "content": "Classifie : 'Mon colis n'arrive pas'" },
    { "role": "assistant", "content": "{" }
  ]
}
```

La réponse commencera par `"intent": ...` directement. Pour fermer proprement, ajoutez `stop_sequences: ["}"]` (et reconcaténez l'accolade fermante côté client) si vous voulez un contrôle maximal.

Pour du **XML strict**, prefillez avec `<answer>` :

```json
{
  "messages": [
    { "role": "user", "content": "Extrais les entités de : ..." },
    { "role": "assistant", "content": "<answer>" }
  ]
}
```

## Solution 3 : Tool Use (le plus robuste)

Pour du JSON garanti structurellement valide, déclarez un **tool** avec un JSON Schema. Claude est forcé de respecter le schéma.

```json
{
  "model": "claude-sonnet-4-6",
  "tools": [
    {
      "name": "classify_message",
      "description": "Classifie un message client",
      "input_schema": {
        "type": "object",
        "properties": {
          "intent": {
            "type": "string",
            "enum": ["question", "commande", "plainte"]
          },
          "confidence": { "type": "number", "minimum": 0, "maximum": 1 },
          "entities": { "type": "array", "items": { "type": "string" } }
        },
        "required": ["intent", "confidence", "entities"]
      }
    }
  ],
  "tool_choice": { "type": "tool", "name": "classify_message" }
}
```

Avec `tool_choice` forcé, Claude **doit** appeler le tool. L'argument `input` du `tool_use` block est garanti valide selon le schéma. C'est le pattern recommandé en production pour les pipelines critiques.

## Markdown structuré

Pour des sorties Markdown destinées à un rendu (docs, emails), instruisez la structure attendue :

```text
Réponds avec exactement cette structure Markdown :

## Résumé
(2-3 phrases)

## Points clés
- (bullet 1)
- (bullet 2)
- (bullet 3)

## Recommandation
(1 paragraphe)
```

Plus la structure est explicite, plus Claude la respectera. Évitez les *"sois clair et bien organisé"* : trop vague.

## Validation côté client (ceinture + bretelles)

Même avec tool use, validez systématiquement avec **Zod**, **Pydantic** ou équivalent. En cas d'échec, retry avec un message qui inclut l'erreur de validation :

```text
Ta réponse précédente a échoué la validation :
{erreur_zod}

Recommence en respectant strictement le schéma.
```

Un single retry corrige 95 % des cas pathologiques restants.

## Pour aller plus loin

- Migrer vos pipelines critiques vers le pattern **Tool Use + `tool_choice` forcé**.
- Mettre en place un wrapper qui log les échecs de parsing pour identifier les prompts à durcir.
- Lire la doc Anthropic *"JSON output"* et *"Tool use"* pour les options avancées (`disable_parallel_tool_use`, etc.).
- Pour les sorties très longues, combiner prefilling + streaming pour réduire la latence perçue.
