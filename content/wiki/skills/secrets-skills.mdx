Un Skill Claude Code est par nature partageable : il finit souvent versionné dans un repo, copié dans `~/.claude/skills/`, publié sur un marketplace interne. Écrire un token Slack ou une clé API dans le `SKILL.md` revient à le diffuser à tout le monde. Ce guide détaille les patterns pour injecter des secrets proprement, isoler les credentials de la logique, et garder un audit trail propre.

## Pourquoi jamais en dur dans le prompt

Trois raisons majeures interdisent les secrets en clair dans un `SKILL.md` :

1. **Le contenu du Skill part dans le contexte Anthropic** : tout ce qui est dans le corps du `SKILL.md` est envoyé à l'API. Un token leak côté logs serveur est possible.
2. **Versioning Git** : un commit qui contient un secret reste dans l'historique même après rewrite — il faut le révoquer immédiatement.
3. **Distribution** : un Skill partagé via marketplace ou plugin embarque le secret pour tous les utilisateurs.

Règle d'or : **le `SKILL.md` ne contient que des références à des variables d'environnement, jamais leurs valeurs.**

## Pattern recommandé : env vars + wrapper script

Le pattern le plus propre consiste à isoler l'appel API dans un script (Bash, Node, Python) qui lit les credentials depuis l'environnement. Le `SKILL.md` ne fait qu'appeler le script.

```md
---
name: slack-notify
description: Send a Slack message to the configured channel.
---

## Usage

Run `./scripts/notify.ts "message"` from this skill directory.

Requires env vars:
- `SLACK_BOT_TOKEN` (xoxb-...)
- `SLACK_CHANNEL_ID`

If either is missing, abort and tell the user to set them in `.env.local`.
```

Côté script, lecture standard :

```ts
const token = process.env.SLACK_BOT_TOKEN;
if (!token) throw new Error("SLACK_BOT_TOKEN manquant");
```

Claude exécute le script via Bash : le token n'apparaît jamais dans le contexte, seul le résultat (succès / erreur) est renvoyé.

## Sources de secrets : ordre de préférence

Du plus sûr au moins sûr :

1. **Gestionnaire de secrets cloud** : Vercel Env Vars, AWS Secrets Manager, Doppler, 1Password CLI. Le script récupère le secret à la volée via la CLI du gestionnaire.
2. **Keychain système** : `security find-generic-password` sur macOS, `secret-tool` sur Linux. Pas de fichier sur disque.
3. **Fichier `.env.local` git-ignored** : pratique en dev, à éviter en équipe.
4. **Variable shell exportée** : OK pour des tests one-shot, perd au reboot.

À bannir : commit dans `.env`, valeur en dur dans un script, copie dans un `SKILL.md`.

## Scoping et révocation

Chaque token utilisé par un Skill doit être **scopé au minimum** :

- Token Slack : un seul scope (`chat:write` pas `admin`)
- Token GitHub : fine-grained, limité aux repos concernés
- Token Jira : compte de service dédié, pas le compte perso

Documenter dans le `SKILL.md` la procédure de révocation : URL exacte, scopes attendus, durée de vie recommandée. Un secret avec rotation à 90 jours et un script de rotation est mille fois mieux qu'un secret éternel.

## Logging et redaction

Les wrappers de Skill peuvent fuiter des secrets dans les logs si on ne fait pas attention. Trois règles :

- Ne jamais logger l'objet `process.env` complet
- Masquer les tokens dans les messages d'erreur (`xoxb-***`)
- Configurer un hook `PreToolUse` qui rejette les commandes contenant des patterns sensibles (`AKIA...`, `ghp_...`, `xoxb-...`)

## Cas particulier : MCP et OAuth

Pour les intégrations OAuth (Google, Slack via app, Notion), préférer un **serveur MCP** qui gère le refresh token côté process MCP, plutôt qu'un Skill qui manipule directement les credentials. Le Skill n'a alors qu'à appeler les outils MCP exposés — aucun secret ne transite par lui.
